# 典型题型与做题心得

这一部分不是泛泛记模板，而是把已经做过的题里真正踩过的坑整理出来。

## 1. 二分：序列二分和答案二分

做题前先分清两件事：
- 序列二分：在有序序列里找位置，二分的是“下标”
- 答案二分：在答案范围里猜答案，二分的是“答案值”
- 它们的共同点都是利用单调性，区别在于单调性来自“序列本身”还是“可行性判断”

### 1.1 序列二分：找到了也不能急着返回

对应题型：查找某个数第一次出现的位置。

核心坑点：
- 当 `a[mid] == x` 时，不能马上返回 `mid`。
- 如果题目要的是“最左边”“最右边”“第一个满足条件的位置”，就必须继续缩区间。

以找最左边位置为例：

```cpp
int find_first(int x, int n) {
    int l = 1, r = n, ans = -1;
    while (l <= r) {
        int mid = l + (r - l) / 2;
        if (a[mid] == x) {
            ans = mid;
            r = mid - 1;
        } else if (a[mid] < x) {
            l = mid + 1;
        } else {
            r = mid - 1;
        }
    }
    return ans;
}
```

心得：
- `mid` 用 `l + (r - l) / 2` 更稳。
- 先想清楚区间含义：当前维护的是答案可能出现的范围。
- 找“左端点”时，命中答案后向左收；找“右端点”时，命中答案后向右收。

补充理解：
- 序列二分的前提是序列本身有序，或者某个判定在下标上单调
- 常见目标不只是“是否存在这个值”，更常见的是找边界
- 例如：第一个 `>= x`、第一个 `> x`、最后一个 `<= x`、最后一个 `< x`
- 如果题目只是问某个值出现次数，往往就是 `upper_bound - lower_bound`

### 1.2 答案二分：先想 `check(mid)` 能不能写

对应题型：
- 求最小可行值
- 求最大可行值
- 最大化最小值、最小化最大值

核心前提：
- 答案范围是有序的
- 能写出一个 `check(mid)`，判断“答案取成 `mid` 是否可行”
- `check(mid)` 的结果要有单调性，否则不能二分

本质：
- 不是在数组里找某个数
- 而是在一段答案区间里，找“可行与不可行的分界点”

常见单调性：
- 求最小可行值时，通常是 `mid` 越大越容易可行
- 求最大可行值时，通常是 `mid` 越小越容易可行

求最小可行值的写法：

```cpp
while (l < r) {
    int mid = l + (r - l) / 2;
    if (check(mid)) r = mid;
    else l = mid + 1;
}
```

求最大可行值的写法：

```cpp
while (l < r) {
    int mid = l + (r - l + 1) / 2;
    if (check(mid)) l = mid;
    else r = mid - 1;
}
```

为什么第二种要写成上取中位数：
- 因为当 `l + 1 == r` 时，如果还写下取中位数，`mid` 会等于 `l`
- 这样在 `check(mid)` 为真时，`l` 不变，循环可能卡住

做题时要先想清楚的三件事：
- 二分的到底是“下标”还是“答案”
- `check(mid)` 判定的是什么
- `check(mid)` 为真后，答案应该往左收还是往右收

心得：
- 很多题看起来不是二分题，但只要答案具有单调性，就可以往答案二分上想
- 答案二分的难点通常不在二分本身，而在 `check(mid)` 怎么设计
- 如果 `check(mid)` 是 `O(n)`，总复杂度通常就是 `O(n log V)`，其中 `V` 是答案范围

## 2. 排序 + 二分统计个数

对应题型：数对统计、相同元素个数统计、离线查询。

典型写法：

```cpp
sort(a, a + n);
int cnt = upper_bound(a, a + n, x) - lower_bound(a, a + n, x);
```

对应心得：
- 先排序，再用二分查找某个值出现了几次。
- `upper_bound - lower_bound` 就是出现次数。

额外注意：
- 计数答案未必装得下 `int`。
- 例如 `n = 1e5` 时，最坏情况数对数可能接近 `1e10`，答案要开 `long long`。

## 3. 单调队列 / 滑动窗口

对应题型：长度固定窗口内的最小值、最大值。

单调队列维护的是“可能成为答案的下标”，不是简单存值。

维护最小值时：
- 队列中的下标对应值要单调递增。
- 新元素进来前，把队尾所有比它大的元素弹掉。
- 窗口滑动后，把已经滑出窗口的队首弹掉。

模板：

```cpp
deque<int> q;
for (int i = 0; i < n; i++) {
    while (!q.empty() && a[q.back()] > a[i]) q.pop_back();
    q.push_back(i);
    while (!q.empty() && q.front() <= i - k) q.pop_front();
    if (i >= k - 1) {
        cout << a[q.front()] << ' ';
    }
}
```

维护最大值时，只需要把比较号反过来：

```cpp
while (!q.empty() && a[q.back()] < a[i]) q.pop_back();
```

心得：
- 队列里存下标，才能判断元素是否已经离开窗口。
- 一次做完最小值后，如果还要做最大值，要记得清空队列。
- 判断过期元素时，常写成 `q.front() <= i - k`。

## 4. 字符串题：完整单词匹配要手写切分

对应题型：统计单词出现次数、忽略大小写、求第一次出现位置。

做法：
1. 先把目标词和整段文本统一转成小写。
2. 按空格切成一个个单词。
3. 逐个和目标词比较，统计次数并记录第一次出现位置。

原因：
- `find` 查的是子串，不是完整单词。
- 题目要求“单词”时，必须考虑边界。

从这类题里得到的坑点：
- 访问字符串前必须确认下标没有越界。
- 手写扫描时，用 `i < s.size()`，不要写成 `i <= s.size()` 后再访问 `s[i]`。
- 文本模拟题宁可步骤写清楚，也不要为了图省事把边界写乱。

## 5. 高精加法

适用场景：
- 两个整数位数很长，超出内置整数范围。

核心思想：
- 把每一位拆开存。
- 从低位到高位逐位相加。
- 用变量记录进位。

模板：

```cpp
string add(string a, string b) {
    vector<int> A, B, C;
    for (int i = (int)a.size() - 1; i >= 0; i--) A.push_back(a[i] - '0');
    for (int i = (int)b.size() - 1; i >= 0; i--) B.push_back(b[i] - '0');

    int carry = 0;
    for (int i = 0; i < (int)max(A.size(), B.size()) || carry; i++) {
        if (i < (int)A.size()) carry += A[i];
        if (i < (int)B.size()) carry += B[i];
        C.push_back(carry % 10);
        carry /= 10;
    }

    string ans;
    for (int i = (int)C.size() - 1; i >= 0; i--) ans += char(C[i] + '0');
    return ans;
}
```

心得：
- 一般把数字倒着存，方便从个位开始算。
- 循环条件里把“还有进位”也写进去，避免漏掉最高位进位。
- 高精题的本质不是难，而是细节多，变量含义必须清楚。

## 6. 批量判素数：先筛再统计

对应题型：
- 需要判断很多个数是否为质数
- 这些待判断的数都落在一个不太大的范围里
- 还可能要求统计不同结果的个数

核心想法：
- 如果候选值很多，就不要对每个数都单独调用判素函数
- 更适合先用筛法一次性预处理，再 `O(1)` 判断某个值是不是质数

这类题里要记住的细节：
- `0` 和 `1` 不是质数，筛法里要单独处理
- 埃氏筛从 `i * i` 开始标记更省重复计算
- 如果题目要求“不同的和”“不同的结果”，可以用 `unordered_set` 自动去重

常见场景：
- 枚举两数和、三数和
- 判断一批候选答案里哪些是质数
- 统计满足条件的不同质数值

心得：
- 这类题真正的优化点通常不在枚举本身，而在“重复判素”改成“预处理后查询”
- 如果题目只关心不同结果，不要自己写复杂去重逻辑，优先想到哈希容器

方法选择：
- 只需要判断少量几个数是不是质数时，优先用普通判素，单次复杂度是 `O(sqrt n)`，空间也最省。
- 需要频繁判断很多个数，而且这些数都落在 `1 ~ n` 的固定范围内时，优先用埃氏筛，先预处理再 `O(1)` 查询。
- 需要筛出 `1 ~ n` 内全部质数，同时还想顺手维护最小质因子、欧拉函数、莫比乌斯函数等信息时，优先用欧拉筛。
- 如果数值范围很大，但真正需要检查的候选很少，不要因为上界大就直接筛到上界，通常“构造候选 + 普通判素”更划算。

快速判断：
- 查询少、数值大：普通判素。
- 查询多、上界固定：埃氏筛。
- 还要附带更多数论信息：欧拉筛。

例子：
- 像回文质数这类题，虽然上界可能很大，但真正需要判断的回文数并不多，更适合先构造候选，再逐个普通判素，而不是直接筛到上界。

## 7. 合法候选很稀疏时，优先枚举候选

对应题型：
- 满足条件的数本身很少
- 整个区间很大，但真正可能成为答案的数很稀疏
- 如果把整个区间都扫一遍，会做很多无用检查

核心想法：
- 不要枚举整个区间，再一个个判断
- 应该直接构造或枚举“合法候选”
- 再对这些候选做检验

常见场景：
- 回文质数
- 特殊进制数
- 满足某种数位结构的数
- 只含某些数字的数

为什么更优：
- 真正可能成为答案的数远少于区间长度
- 把“区间枚举”改成“候选枚举”后，复杂度通常会大幅下降

做题提醒：
- 先问自己：答案有没有明显结构
- 如果有，就先想能不能直接构造这些数
- 不要一看到区间题就默认从 `L` 枚举到 `R`

## 8. 区间筛：关键在起点怎么取

对应题型：
- `L`、`R` 很大，但区间长度 `R - L` 不大
- 只需要求出区间 `[L, R]` 内哪些数是质数，或者统计个数

核心想法：
- 先筛出 `1 ~ sqrt(R)` 内的所有质数
- 再用这些质数去筛区间 `[L, R]`
- 区间里没被标记的位置就是质数

最关键的细节：
- 对每个质数 `p`，起点不是随便取的
- 区间内第一个 `p` 的倍数是 `ceil(L / p) * p`
- 但真正开始筛时，要取 `max(p * p, ceil(L / p) * p)`

为什么还要和 `p * p` 取最大值：
- 如果某个倍数小于 `p * p`，那它可以写成 `p * k`，并且 `k < p`
- 这说明它含有更小的因子，应该更早被更小的质数筛掉
- 这样还能避免把质数 `p` 自己误标成合数

常见坑点：
- 别忘了把区间里的 `0` 和 `1` 单独标成非质数
- `ceil(L / p)` 常写成 `(L + p - 1) / p`
- `sqrt(R)` 最好做一次上下微调，避免浮点误差

## 9. 在线维护前缀信息

对应题型：
- 数据一边输入，一边要求输出当前前缀的统计结果
- 每次都要基于“前面所有数”计算一个答案
- 如果每轮都重新扫一遍前缀，复杂度会白白变大

核心想法：
- 把“每次重新统计前缀”改成“边读边维护需要的信息”
- 前缀和、当前最大值、当前最小值这类量都适合增量更新

常见维护量：
- 前缀和：`sum += x`
- 当前最大值：`mx = max(mx, x)`
- 当前最小值：`mi = min(mi, x)`
- 当前元素个数：随下标或计数器一起维护

典型场景：
- 输出当前前缀平均值
- 去掉最大值和最小值后的平均分
- 前缀最值、前缀贡献、滚动统计量

心得：
- 只要题目在不断扩展前缀，就优先想“在线维护”
- 前一个状态如果已经算出来了，新加入一个数通常只会改动少数几个统计量
- 这类题的关键不是复杂公式，而是别把已经算过的前缀信息重复计算

### 9.1 从二重枚举改成边扫边统计

对应题型：
- 找满足条件的下标对
- 找满足条件的切分对
- 找“前一个位置”和“后一个位置”的合法配对数

原始做法常是：
- 枚举第一刀
- 枚举第二刀
- 判断这对切法是否合法

这样本质上是在枚举所有二元组，复杂度通常是 `O(n^2)`。

优化后的核心想法：
- 只枚举后一个位置
- 同时维护“前面有多少个位置可以当第一个位置”
- 当前点一旦能当后一个位置，就把前面所有合法位置的个数直接加进答案

这类优化的本质是：
- 把“枚举所有二元组”
- 改成“扫描当前点时，利用前面已经统计好的信息”

也就是把当前贡献写成：

```text
当前贡献 = 当前点能和前面多少个点配对
```

例如切成三段且三段和相等时：
- 第一刀合法：前缀和等于 `t`
- 第二刀合法：前缀和等于 `2 * t`

扫描到某个位置 `i` 时：
- 如果 `pre[i] == 2 * t`，说明它可以当第二刀，就把前面所有能当第一刀的位置数加到答案里
- 如果 `pre[i] == t`，说明它可以作为未来的第一刀，就把计数器加一

典型写法：

```cpp
if (pre[i] == 2 * t) ans += cnt;
if (pre[i] == t) cnt++;
```

做题时要记住的判断：
- 如果题目是在找“两个位置”的配对数，不要急着写双重枚举
- 先想条件能不能拆成“前面哪些点合法”加“当前点是否能作为后一个点”
- 如果能，就常常可以只扫一遍，同时维护前面合法位置个数

心得：
- 很多 `O(n^2)` 的配对统计，本质都能改写成“枚举后项 + 统计前项”
- 一旦题目问的是“当前点和前面多少点能组成答案”，就要立刻想到这种写法
- 这类优化的关键不是少写循环，而是把“配对判断”变成“前缀统计”

## 10. 贡献化后排序贪心

对应题型：
- 最终比较的是几个总和之间的大小
- 原式看起来有多组数据、多维度比较
- 但每个位置对答案的影响可以独立拆开

核心想法：
- 先把判定式移项
- 把每个位置变成一个“贡献值”
- 原问题就会从多维比较，压成一维选数问题

例如原式：

```text
X > Y + Z
```

可以改写成：

```text
X - Y - Z > 0
```

于是每个位置的贡献就是：

```text
contribution[i] = x[i] - y[i] - z[i]
```

接下来常见问题会变成：
- 选尽量多的数，使总和仍然大于 0

这时要想到：
- 如果固定选 `k` 个，怎样让总和最大？
- 答案一定是选最大的 `k` 个贡献值

所以典型做法就是：
- 先算贡献
- 再降序排序
- 再扫前缀和

贪心依据：
- 前 `k` 大的和，是所有“选 `k` 个”的方案里最大的
- 如果前 `k` 大都不能满足条件，别的 `k` 个更不可能满足

做题提醒：
- 注意判定条件到底是 `> 0` 还是 `>= 0`
- 如果前缀和一旦不合法，后面通常也不用继续扩大答案
- 如果有多种阵营 / 多种目标，就分别算贡献后各做一遍

## 11. 暴力题也要先估复杂度

从已有代码里也能看出一个共性：
- 写法能跑，不代表复杂度合理。
- 每次看到双重循环、反复删除容器元素、反复线性找最大值，都要先问一句最坏复杂度是多少。

比赛时最常见的失误不是不会写，而是：
- 没先估复杂度
- 没发现答案会爆 `int`
- 没处理边界

## 12. 排序后二分：固定中间项，左右数量相乘

对应题型：
- 递增三元组计数
- 固定一个中间值，统计左右两边有多少合法选择
- 形如 `a < b < c` 的计数题

核心想法：
- 不要直接三重枚举
- 把中间项固定住
- 分别统计左边能选几个、右边能选几个
- 当前中间项的贡献就是两边方案数相乘

例如：
- 对每个 `b`
- 统计 `A` 中有多少数 `< b`
- 统计 `C` 中有多少数 `> b`
- 那么以这个 `b` 为中间项的合法三元组个数就是：

```text
左边数量 × 右边数量
```

排序后二分统计：

```cpp
long long left_cnt = lower_bound(A.begin(), A.end(), b) - A.begin();
long long right_cnt = C.end() - upper_bound(C.begin(), C.end(), b);
ans += left_cnt * right_cnt;
```

为什么成立：
- 左边的选法和右边的选法彼此独立
- 乘法原理：左边选一个、右边选一个，就组成一种方案

这类题的本质：
- 把三元组计数拆成“枚举中间项”
- 再把另外两维变成独立计数

做题提醒：
- `lower_bound` 统计的是 `< b` 的个数
- `upper_bound` 配合尾指针，统计的是 `> b` 的个数
- `left_cnt * right_cnt` 可能爆 `int`，中间结果要开 `long long`

## 13. 相邻增量贪心：答案只来自比前一个位置多出来的部分

对应题型：
- 从左到右逐步覆盖一段序列
- 前面的操作能自然覆盖后面一部分
- 只有“比前一个位置更高”的部分才需要新增代价

典型想法：
- 从左往右扫
- 如果当前值不超过前一个值，说明前面的操作还能继续覆盖
- 如果当前值更大，新增的代价就是这段“多出来的高度”

典型写法：

```cpp
ans = a[1];
for (int i = 2; i <= n; i++) {
    if (a[i] > a[i - 1]) ans += a[i] - a[i - 1];
}
```

也可以统一写成：

```cpp
ans = 0;
for (int i = 1; i <= n; i++) {
    ans += max(0, a[i] - a[i - 1]);
}
```

其中可视为：

```cpp
a[0] = 0;
```

为什么这样算：
- 前面已经铺过的层数，对后一个位置仍然有效
- 只有当前比前一个更高时，才需要新开操作去补这部分高度

这类题的本质：
- 不是看总和
- 而是看“相邻位置之间新增了多少需求”

常见场景：
- 铺路
- 逐段加高
- 连续区间操作能向右延续的贪心题

做题提醒：
- 先想清楚前一位置的操作能不能延续到当前位置
- 如果能，答案往往和“正的相邻差”有关
- 注意边界：第一个位置相当于和 `0` 比较

## 14. 数轴分类讨论：先列候选路径，再比较最小代价

对应题型：
- 数轴上有多个关键点
- 行走顺序不同会导致总代价不同
- 需要比较“先去 A 再去 B”还是“先去 B 再去 A”

核心想法：
- 不要一上来就陷进很多符号讨论
- 先把可能的策略写成几个候选答案
- 每个候选答案都直接算路程
- 最后取最小值

例如常见候选：
- 先去拿钥匙，再去开箱
- 先去箱子处处理一部分，再去拿钥匙回来

这类题真正重要的是：
- 把问题拆成“几种可能路线”
- 每条路线的代价写成清楚的距离表达式

做题提醒：
- 数轴题优先想到绝对值距离 `abs(x - y)`
- 先把“会不会经过某点”“需不需要折返”想清楚
- 与其写很多层 `if`，不如先列出有限个候选方案再 `min`
