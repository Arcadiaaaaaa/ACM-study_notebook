# 常见技巧与典型思想

## 1. 方向数组

作用：把网格中“相邻位置”的变化量统一存起来，便于循环处理。

八方向：

```cpp
int dx[8] = {-1, -1, -1, 0, 0, 1, 1, 1};
int dy[8] = {-1, 0, 1, -1, 1, -1, 0, 1};
```

示例：

```cpp
for (int k = 0; k < 8; k++) {
    int nx = x + dx[k];
    int ny = y + dy[k];
    if (0 <= nx && nx < n && 0 <= ny && ny < m && isboom(nx, ny)) {
        cnt++;
    }
}
```

注意：
- 使用前先统一坐标含义。
- 越界判断要写全。

## 2. 规则表 / 映射表

适用场景：
- 题目规则本身就是一一对应关系。

示例：

```cpp
beat["elephant"] = "tiger";
if (beat.count(a) && beat[a] == b) {
    // a 能赢 b
}
```

注意：
- 如果 `a` 可能不存在，先判断 `count`，避免误插入默认值。

## 3. 编号后利用取余找规律

适用场景：
- 多个对象构成环形关系。

示例：

```cpp
// elephant = 0, tiger = 1, cat = 2, mouse = 3
// x 吃 y
y == (x + 1) % 4
```

核心思想：
- 先编号。
- 再把题目的循环关系转成模运算。

## 4. 约瑟夫问题

关键思想：
- 删除一个人后，对剩余元素重新编号。
- 把原问题转化为规模减一的同类问题。

本质：
- 下标映射
- 重编号
- 规模缩小

## 5. 日期模拟

适用场景：
- 日期推进
- 计算若干天后的日期
- 判断两个日期之间相隔多少天
- 日历题、签到题、时间模拟题

### 5.1 闰年定义

闰年的定义是：
- 能被 `400` 整除
- 或者能被 `4` 整除但不能被 `100` 整除

写法：

```cpp
bool isLeap(int year) {
    return (year % 400 == 0) || (year % 4 == 0 && year % 100 != 0);
}
```

记忆：
- 普通年份里，2 月有 `28` 天
- 闰年里，2 月有 `29` 天
- 也就是：2 月比平年多一天

### 5.2 每个月多少天

平年可以先写成：

```cpp
int days[13] = {0, 31, 28, 31, 30, 31, 30, 31, 31, 30, 31, 30, 31};
```

如果当前年份是闰年，就把 2 月改成 `29`：

```cpp
days[2] = isLeap(year) ? 29 : 28;
```

### 5.3 常见日期推进写法

例如把日期往后推一天：

```cpp
day++;
if (day > days[month]) {
    day = 1;
    month++;
    if (month > 12) {
        month = 1;
        year++;
    }
}
```

核心顺序：
- 先处理天
- 天越界后处理月
- 月越界后处理年

### 5.4 做题提醒

- 日期题最容易错的是 2 月
- 写月份天数时，先写平年数组，再单独特判闰年
- 只要涉及跨年、跨月，就不要靠脑算，优先老老实实模拟
- 如果题目要求很多次推进日期，优先先把“判断闰年”和“每月天数”写成函数

## 6. 二维前缀和

适用场景：
- 多次查询某个子矩阵元素和
- 矩阵固定不变，但查询很多
- 想把每次 `O(nm)` 的区域求和降到 `O(1)`

核心思想：
- 一维前缀和是“前 `i` 个数的和”
- 二维前缀和是“从左上角 `(1,1)` 到 `(i,j)` 的整个矩形和”

通常定义：

```cpp
s[i][j] = a[1][1] 到 a[i][j] 这个矩形的元素和
```

递推公式：

```cpp
s[i][j] = s[i - 1][j] + s[i][j - 1] - s[i - 1][j - 1] + a[i][j];
```

为什么要减掉 `s[i - 1][j - 1]`：
- 因为它在前两项里被重复算了一次

模板：

```cpp
for (int i = 1; i <= n; i++) {
    for (int j = 1; j <= m; j++) {
        s[i][j] = s[i - 1][j] + s[i][j - 1] - s[i - 1][j - 1] + a[i][j];
    }
}
```

查询子矩阵 `(x1, y1)` 到 `(x2, y2)` 的和：

```cpp
sum = s[x2][y2] - s[x1 - 1][y2] - s[x2][y1 - 1] + s[x1 - 1][y1 - 1];
```

理解方式：
- 先取大矩形 `s[x2][y2]`
- 减去上面多算的一块
- 减去左边多算的一块
- 左上角被减重了，再补回来一次

做题提醒：
- 最好统一用 1 下标，这样公式最整齐
- 数组常开成 `n + 1` 行、`m + 1` 列，方便处理第 `0` 行和第 `0` 列
- 先统一坐标含义：通常 `i` 表示行，`j` 表示列
- 如果矩阵值大、矩形和也可能很大，前缀和要开 `long long`

## 7. 二维差分

适用场景：
- 多次给某个子矩阵整体加上一个值
- 修改很多，最后再统一求结果
- 想把每次区域修改从 `O(nm)` 降到 `O(1)`

核心思想：
- 一维差分是“记录变化的起点和终点”
- 二维差分是“记录一个矩形对四个角的影响”

如果要给矩形 `(x1, y1)` 到 `(x2, y2)` 全部加上 `c`，则：

```cpp
b[x1][y1] += c;
b[x2 + 1][y1] -= c;
b[x1][y2 + 1] -= c;
b[x2 + 1][y2 + 1] += c;
```

含义是：
- 从左上角开始加
- 在矩形下方消掉影响
- 在矩形右侧消掉影响
- 在右下角把重复消掉的部分补回来

常用封装：

```cpp
void insert(int x1, int y1, int x2, int y2, int c) {
    b[x1][y1] += c;
    b[x2 + 1][y1] -= c;
    b[x1][y2 + 1] -= c;
    b[x2 + 1][y2 + 1] += c;
}
```

所有修改做完后，再还原最终矩阵：

```cpp
for (int i = 1; i <= n; i++) {
    for (int j = 1; j <= m; j++) {
        b[i][j] += b[i - 1][j] + b[i][j - 1] - b[i - 1][j - 1];
    }
}
```

这里还原时用的，其实也是二维前缀和思想。

如果原矩阵一开始就有值：
- 可以把每个 `a[i][j]` 看成对矩形 `(i, j, i, j)` 加上 `a[i][j]`
- 也就是先对每个点做一次 `insert(i, j, i, j, a[i][j])`
- 然后再处理题目给出的矩形修改

做题提醒：
- 仍然建议统一用 1 下标
- 差分数组通常要多开一圈，避免访问 `x2 + 1`、`y2 + 1` 越界
- 如果题目只改不查，差分通常比每次暴力修改高效得多
- 如果题目既频繁修改又频繁查询，就不能只靠普通前缀和 / 差分，往往要考虑树状数组、线段树一类结构

## 8. 轮次标记法：用 `tag` 代替反复清空 `vis`

适用场景：
- 外层一轮一轮枚举左端点
- 每一轮都要重新统计“这个值在本轮是否出现过”
- 如果每轮都 `memset(vis, 0, sizeof vis)`，会产生很多重复开销

典型原型是：
- 枚举左端点 `l`
- 向右枚举右端点 `r`
- 需要判断 `a[r]` 在这一轮里是不是第一次出现

朴素想法通常是：

```cpp
for (int l = 1; l <= n; l++) {
    memset(vis, 0, sizeof vis);
    for (int r = l; r <= n; r++) {
        if (!vis[a[r]]) {
            vis[a[r]] = 1;
            // 第一次出现
        }
    }
}
```

问题在于：
- 逻辑上每一轮确实要“清空”
- 但不一定真的要把整个数组重新赋成 `0`

优化思想：
- 给每一轮枚举分配一个唯一编号 `tag`
- `vis[x]` 不再只存 `0 / 1`
- 而是存“这个值上一次在哪一轮出现过”

于是：
- 如果 `vis[x] == tag`，说明 `x` 在当前这一轮已经出现过
- 如果 `vis[x] != tag`，说明 `x` 在当前这一轮还没出现过

典型写法：

```cpp
int tag = 0;
for (int l = 1; l <= n; l++) {
    tag++;
    for (int r = l; r <= n; r++) {
        int x = a[r];
        if (vis[x] == tag) continue;
        vis[x] = tag;
        // x 在这一轮第一次出现
    }
}
```

这句的本质是：

```text
vis[x] == tag
```

不表示“这个值永远出现过”，而表示：

```text
这个值在当前轮次里已经出现过
```

为什么这样能省掉清空数组：
- 因为旧轮次留下来的标记，轮次编号已经不同了
- 只要 `tag` 变了，旧数据就自动“失效”
- 相当于用“当前轮次编号”把不同轮的访问记录隔开了

这类优化的本质：
- 不是清空状态
- 而是给状态打上“属于哪一轮”的时间戳

常见场景：
- 枚举每个左端点，统计不同元素个数
- 枚举区间起点时，对当前轮做去重
- 多次独立搜索 / 多次独立统计时，避免反复清空标记数组

什么时候特别值得用：
- `vis` 数组很大
- 外层轮数很多
- 每轮真正访问到的值远少于数组大小

使用前提：
- 值 `x` 能直接当数组下标，或者已经离散化
- 如果值域特别大且稀疏，可能更适合 `unordered_map`

做题提醒：
- `tag` 每轮都要加一，不能忘
- 如果轮数特别多，要注意 `tag` 的类型，通常用 `int` 就够，大到担心溢出时可改 `long long`
- 这种写法优化的是“清空数组”的代价，不会改变内层真正访问元素的复杂度

记忆方式：
- `vis[x]` 存的不是“是否访问过”
- 而是“上次在哪一轮访问过”

## 9. 连续相同段问题：用双指针按段扫描

适用场景：
- 需要处理一行 / 一列里的连续相同元素
- 要找长度至少为 `k` 的连续段
- 想避免一格一格重复判断同一段

典型题型：
- 消除类游戏
- 统计连续相同字符段
- 压缩连续段
- 按段做区间标记

核心思想：
- 令 `j` 指向当前连续段起点
- 令 `k` 向右扩展，找到这一段的第一个不同位置
- 区间 `[j, k - 1]` 就是一整段相同元素
- 处理完这一整段后，直接跳到 `j = k`

典型写法：

```cpp
int j = 0;
while (j < m) {
    int k = j + 1;
    while (k < m && a[i][k] == a[i][j]) k++;

    if (k - j >= 3) {
        for (int p = j; p < k; p++) {
            mark[i][p] = true;
        }
    }

    j = k;
}
```

为什么这样更好：
- 每一段只会被完整扫描一次
- 不会在同一段内部重复从头扩展
- 思路也比“枚举每个点再往两边试”更清楚

如果题目要求“同时生效”：
- 行扫描时先只做标记
- 列扫描时也只做标记
- 全部扫描完后，再统一处理结果

这一步很关键：
- 不要一边扫描一边直接修改原数组
- 否则会影响后续判断，破坏“同时消除”的含义

常见场景变形：
- 扫每一行找连续段
- 扫每一列找连续段
- 先标记，再统一清零 / 删除 / 计数

做题提醒：
- 处理完一整段后，记得写 `j = k`，不是 `j++`
- 如果既扫行又扫列，坐标含义要始终统一
- 标记数组和原数组最好分开

## 10. 矩阵变换题：把操作写成函数后枚举候选

适用场景：
- 旋转矩阵
- 镜像矩阵
- 判断一个图案是否能经过若干固定变换变成另一个图案

核心思想：
- 不要在主逻辑里手写一大堆下标推导
- 先把“旋转 90 度”“镜像”各自写成函数
- 再把所有可能的候选变换枚举出来比较

例如：

```cpp
vector<string> rotate90(vector<string> a);
vector<string> reflect(vector<string> a);
```

主逻辑就可以写成：

```cpp
if (rotate90(a) == b) ...
if (rotate90(rotate90(a)) == b) ...
if (reflect(a) == b) ...
```

这样做的好处：
- 每个变换单独调试，出错更容易定位
- 主流程变成“枚举候选答案”，逻辑清楚
- 不容易把不同变换的下标关系写混

这类题的本质：
- 不是复杂数学推导
- 而是把每个变换当成一个确定函数
- 然后枚举所有允许的变换结果

做题提醒：
- 先统一矩阵下标含义：`i` 行、`j` 列
- 新矩阵大小一般和原矩阵相同
- 先写小样例手推一次旋转和镜像，再落代码
- 如果容器支持直接比较，如 `vector<string>`，可以直接用 `==`

## 11. 贡献法

适用场景：
- 最终答案是很多位置、很多区间、很多选择的总和
- 直接按题意整体统计很难
- 但每个元素对答案的影响可以单独拆开

核心思想：
- 不直接问“整个答案怎么求”
- 改问“某个元素 / 某个位置 / 某种选择，对答案贡献了多少”
- 再把所有贡献加起来

本质是把：

```text
整体问题
```

拆成：

```text
答案 = 各部分贡献之和
```

常见形式：
- 每个位置贡献多少次
- 每个值被选中时给答案加多少
- 每个区间包含某个点时会带来多少增量
- 每个中间项能和左右多少方案配对

为什么好用：
- 原题往往是“整体不好算，局部好算”
- 一旦每个部分的贡献能单独表达，复杂度通常会明显下降

常见写法方向：

```text
ans = Σ contribution[i]
```

或者：

```text
ans += 当前元素的贡献
```

典型例子：
- 递增三元组里，固定中间项后，当前 `b` 的贡献是：

```text
左边比它小的个数 × 右边比它大的个数
```

- 贡献化贪心里，把原判定式移项后，每个位置变成一个独立贡献值：

```text
contribution[i] = x[i] - y[i] - z[i]
```

- 相邻增量类题里，当前位置真正新增的贡献常常是：

```text
max(0, a[i] - a[i - 1])
```

做题时要问自己的几句话：
- 能不能把答案写成“每个位置贡献多少”的和
- 能不能固定一个点，再算它和别的位置能组成多少方案
- 能不能把全局条件移项，变成单个元素的贡献值

常见好处：
- 让三维、二维问题降成一维累计
- 让不好直接计数的问题转成独立计数
- 常和排序、前缀和、二分、贪心一起出现

做题提醒：
- 贡献可能会重复算，要先想清楚“每种情况是否只被统计一次”
- 贡献值可能是负数，也可能是方案数
- 乘法型贡献、中间结果常常要开 `long long`

记忆方式：
- 不要老想着“答案整体怎么求”
- 先问“每个东西到底给答案贡献了什么”

## 12. 递推维护

适用场景：
- 当前状态和前一个状态关系很强
- 每次只新增一个元素 / 一步变化
- 如果每次从头重算，会做很多重复计算

核心思想：
- 已经算出 `i - 1` 的结果后
- 不要重新算 `i`
- 而是只根据“新增的那一点变化”把它维护出来

本质是把：

```text
每次从头重算
```

改成：

```text
用前一个结果递推到当前结果
```

常见形式：

```text
f(i) = f(i - 1) + 本轮新增影响
```

或者：

```text
当前统计量 = 上一轮统计量 + / - 少量变化
```

为什么好用：
- 前一轮已经包含了绝大多数信息
- 当前轮通常只比前一轮多一个点、少一个点、改动一个局部
- 所以真正需要重新算的内容很少

典型例子：
- 前缀和：

```cpp
sum[i] = sum[i - 1] + a[i];
```

- 在线维护最值：

```cpp
mx = max(mx, x);
mi = min(mi, x);
```

- 连续阶乘中因子个数维护：

```text
f(i) = f(i - 1) + cnt5(i)
```

- 配对统计时的计数器维护：

```cpp
if (pre[i] == t) cnt++;
```

这类题的判断方式：
- 当前答案能不能由上一个答案推出来
- 新加入一个元素后，到底只会影响哪些量
- 有哪些信息可以滚动维护，而不是重复计算

和 DP 的关系：
- 递推维护不一定是正式 DP
- 但它和 DP 共享一个核心：复用前面已经算出的结果
- 区别只是很多时候这里只维护几个统计量，而不是一整张状态表

常见场景：
- 前缀统计
- 滚动维护答案
- 连续序列处理
- 枚举过程中边走边更新计数器

做题提醒：
- 先想清楚维护量的含义，别让变量语义混乱
- 如果维护的是答案或中间和，注意是否会溢出
- 一旦发现“每轮都在重复扫前面”，就要立刻想递推维护

记忆方式：
- 前一个结果如果能复用，就不要重算
- 关键不是公式长不长，而是“新增一步到底改了什么”
